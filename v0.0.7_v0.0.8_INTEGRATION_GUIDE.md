   ```javascript
     const accessToken = jwt.sign(payload, SECRET, { expiresIn: "1h" })
     const refreshToken = jwt.sign(payload, REFRESH_SECRET, { expiresIn: "7d" })

     // Verify refresh token signature
     const payload = jwt.verify(token, REFRESH_SECRET)
     
     // Check if token still in DB (not revoked)
     const dbToken = await RefreshToken.findOne({ token })
     if (!dbToken) throw new Error("Token revoked")
     
     // Issue new access token
     return jwt.sign(payload, SECRET, { expiresIn: "1h" })
   }
   ```

## 3️⃣ v0.0.7 - LABOR DISTRIBUTION ENGINE

### Objetivo
Cerrar el loop con trabajadores: cada empleado ve cómo se distribuye el excedente y recibe su cuota transparentemente.

### Arquitectura

```
Period (has surplus $1000)
  │
  ├─→ Distribution Rule (e.g., "proportional-salary")
  │
  ├─→ Distribution Service
  │    ├─ Get employees + their salaries
  │    ├─ Apply rule: (salary / totalSalary) * surplus
  │    ├─ Calculate each employee's share
  │    └─ Return breakdown
  │
  └─→ Period.distribution = {
        byEmployee: {
          "emp-123": { fixed: 0, variable: 400, total: 400 },
          "emp-456": { fixed: 0, variable: 600, total: 600 }
        },
        rule: "proportional-salary",
        totalDistributed: 1000,
        calculatedAt: Date,
        auditHash: SHA256(...)
      }
```

### User Flows

#### Admin: Distribute Surplus
```
1. Admin goes to Period Management
2. Selects May 2026 period
3. Sees: Surplus calculated = $1,000
4. Clicks "DISTRIBUIR SEGÚN REGLA"
5. System runs distribution algorithm
6. Shows breakdown: Employee A: $400, Employee B: $600
7. Confirms
8. Distribution locked (read-only, can appear in statements)
```

#### Employee: View My Share
```
1. Employee logs in
2. Goes to "Mi Distribución"
3. Sees his May 2026 share: $400
4. Breakdown: "40% of $1000 surplus (because my salary is 40% of payroll)"
5. Full transparency: can see calculation
6. Can download statement
```

### Algorithms to Implement

1. **Proportional to Salary** (default)
   - Formula: `(employee.salary / totalSalaries) * surplus`
   - Example: Juan $1000, Maria $2000, Surplus $900 → Juan $300, Maria $600

2. **Proportional to Productivity**
   - Uses manager-assigned score (0-100)
   - Formula: `(employee.score / totalScores) * surplus`

3. **Fixed Equal**
   - Same amount for everyone: `surplus / employeeCount`

4. **Fixed + Variable**
   - Each gets $X fixed, remainder distributed by metric

## 4️⃣ v0.0.8 - CERTIFICATION LAYER

### Objetivo
Congelar períodos: una vez certificado, NINGÚN dato puede cambiar. Esto es punto de no retorno para legal compliance.

### Arquitectura

```
After Distribution Complete:
  │
  Period has { result, distribution }
  │
  ├─→ Run Pre-Certification Audit
  │    ├─ ✓ Result calculated?
  │    ├─ ✓ Distribution calculated?
  │    ├─ ✓ All employees covered?
  │    ├─ ✓ Sum matches surplus?
  │    └─ ✓ No anomalies?
  │
  ├─→ (If audit passes) Create snapshots:
  │    ├─ inputsSnapshot: {sales, costs, employees}
  │    ├─ outputsSnapshot: {surplus, distribution}
  │
  ├─→ Calculate hashes:
  │    ├─ inputsHash = SHA256(inputsSnapshot)
  │    ├─ outputsHash = SHA256(outputsSnapshot)
  │
  ├─→ Create Certificate:
  │    ├─ number: CERT-2026-05-001
  │    ├─ inputsHash, outputsHash
  │    ├─ locks period (locked: true)
  │
  └─→ Period is NOW FROZEN
       └─ Any attempt to modify: REJECTED
```

### Security Properties

1. **Immutability**: Once locked, data cannot change
2. **Integrity**: SHA256 hashes detect any modification
3. **Verification**: Public endpoint verifies certificate
4. **Audit Trail**: When/who issued, expiration, reason

### Scenario: Tampering Detection

```
Scenario: Someone tries to change distribution after certification

1. Period is certified with outputsHash = ABC123...
2. Hacker modifies Period.distribution.byEmployee["emp-123"].total
3. Admin requests: GET /api/certifications/verify/period-id
4. System recalculates outputsHash with current data
5. New hash = XYZ789... (different!)
6. Alert: ⚠️ CERTIFICATE TAMPERING DETECTED
7. Admin is notified immediately
8. Certificate marked as invalid
9. All history logged with timestamp + who modified what
```

## 7️⃣ RISK MITIGATION

### Risk: Race Conditions (v0.0.7-v0.0.8)

**Problem**: Two users try to distribute same period simultaneously

**Solution**: 
```javascript
// Use MongoDB transactions + version locking
const session = await mongoose.startSession()
session.startTransaction()

try {
  // Find with lock
  const period = await Period.findOne(
    {_id, companyId, distributionLocked: false},
    {},
    {session}
  )
  
  if (!period) throw new Error("Already being distributed")
  
  // Do work
  const dist = calculateDistribution(period)
  
  // Update with lock
  await Period.updateOne(
    {_id},
    {distribution: dist, distributionLocked: true},
    {session}
  )
  
  await session.commitTransaction()
} catch (error) {
  await session.abortTransaction()
  throw error
}
```

### Risk: Hash Collision

**Probability**: ~1 in 2^256 (effectively zero)  
**Mitigation**: Use SHA256 (industry standard for 20+ years)

### Risk: Period Modification After Lock

**Solution**: API rejects any Period.update() where locked=true
```javascript
// Controller
if (period.locked) {
  return sendError(res, 403, "Period is locked")
}
```
